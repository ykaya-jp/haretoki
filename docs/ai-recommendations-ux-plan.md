# AI レコメンド UX 再設計計画

作成: 2026-04-14 / 担当: Design + PM / スコープ: `/explore` の「AIおすすめ式場」セクション

## 1. 現状の動作（事実ベース）

対象: `src/components/venues/ai-recommendations.tsx`（Client Component、`explore/page.tsx:129` に常時マウント）

フロー:

1. マウント時 `useEffect` で `sessionStorage("ai-recs-v1")` を読む（TTL 1h）。
2. キャッシュヒット: 即座に結果を state に反映し API を叩かない。
3. キャッシュミス: `loading=true` → gold-subtle プレースホルダ「あなたの条件に合う式場を探しています...」を描画 → `getOnboardingRecommendations()` (Server Action) で Claude 呼び出し。
4. Server Action は `isClaudeAvailable()` が false、JSON parse 失敗、例外のいずれでも **silent に `null`** を返す。
5. `recommendations` が null / [] の場合、コンポーネントは **`return null`** で丸ごと消える。

条件ゲート: **なし**。Venue 登録 0 件でも Claude を叩く。既存 Venue 名はプロンプトの除外ヒントに渡すだけ。プロジェクトの `conditions` がなくても「デフォルトで」推薦を生成する。

## 2. ユーザーが体験している問題（再定義）

| # | 症状 | 根本原因 |
|---|---|---|
| A | **ちらつき** 一瞬出て消える | `loading=true` で placeholder を描画 → レスポンス失敗/空 → `return null` で DOM ごと消失。キャッシュ初回時は無音で Claude 呼び出しが走るため SSR/CSR の可視状態差分も発生 |
| B | **不透明** 何を根拠に推薦するか不明 | `advice` は任意文字列。入力条件（エリア/予算/ゲスト数）がカードに 1 件も可視化されていない |
| C | **条件ゲート非明示** | コード上にゲート自体がない。ユーザーは「なぜ今動かないのか」を推測できない |
| D | **価値未形成** | セクション見出しが「AIおすすめ式場」だけ。何をしてくれる機能か、どのくらい信用していいかが分からない |
| E | **コントロール欠如** | 閉じる / 非表示 / フィードバックの導線なし。押し付け感 |
| F | **失敗の沈黙** | Claude 失敗時も網羅的に `null`。ユーザーには"出ない時が機能してない時"にしか見えない |

## 3. ゴールと指標

- **透明性**: 1 行で「今の条件と既存候補 N 件から提案しています」が読める。
- **邪魔しない**: データ不足・失敗時は "空気化" せず、**予告**か**理由**を控えめに示す。
- **信頼**: 候補には必ず根拠チップ（条件との一致、予算内、エリア一致）を 1 つ以上表示。
- **測定**:
  - 推薦カード表示率 / CTR
  - 「追加」押下 → D0 式場追加率への寄与（A/B）
  - Sentry: `AIRecommendations` マウント後 2s 以内の DOM 消失イベント計測（ちらつき検知）
  - Claude Server Action の失敗率・p95 レイテンシ

## 4. 対策

### A. トリガー条件の再設計（最重要）

- 登録 Venue 数に応じ**3 状態**に明示分岐。Claude 呼び出しは `>=1` の時だけ。
  - `venues=0`: Claude を呼ばず **「期待値予告カード」** を出す。見出し「まずは 1 件、気になる式場を追加」本文「追加した式場の傾向からAIがおすすめを提案します」CTA「URLから追加」。
  - `venues>=1 かつ conditions>=1 項目`: 通常推薦。
  - `conditions 空 かつ venues=0`: 非表示（プレゼンスなし。ここだけ沈黙を許す）。
- `isClaudeAvailable()` が false の時はサーバー側で `{ state: "disabled" }` を返し、クライアントでは**何も描画しない**（冪等な非表示。ちらつき根絶）。

### B. 状態別 UI の明示化（5 状態）

| State | 文言 | 視覚 |
|---|---|---|
| Primed（予告） | 「気になる式場を追加すると、AI が次の 1 件を提案します」 | gold icon + 薄い gold-subtle 帯、CTA 1 本 |
| Loading | 「おふたりの条件（◯◯エリア・◯万円以内）で候補を整えています」 | スケルトン 2 枚 + 条件チップ表示 |
| Result | タイトル「あなたの希望に寄り添う 3 件」サブ「◯◯を重視 / ◯万円以内 / 登録済み N 件を除外」 | カードに根拠チップ必須 |
| Empty-Result（成功したが 0 件） | 「今の条件では新しい提案が見つかりませんでした。条件を広げてみますか？」 | 条件編集へのリンク |
| Error | 「いま少し混雑しています。少し後でまたお試しください」＋再試行ボタン | 赤系は使わず muted |

**重要**: Loading → Error/Empty 遷移時は `return null` せず、**同じ高さの説明枠**を保つ。これでちらつきは物理的に消える。

### C. "裏で動いてそう" を解消

- `useEffect` 即発火をやめ、**Server Component でサーバーサイド初期状態**を計算（`venues=0` → Primed、`conditions` なし → 非表示）。Claude 呼び出しだけ Client で遅延実行。これで SSR 時点で最終的な枠の有無が確定し、hydration 後の flicker が消える。
- プレースホルダ高さを最小 88px に固定（CLS 防止）。
- 「閉じる」押下時は `sessionStorage("ai-recs-hidden-until", Date.now()+24h)` で 1 日非表示。ブランドコピー「今日はここまで。あしたの朝にまた。」

### D. 推薦ロジックの可視化（「なぜ?」）

- セクション直下に折りたたみ `details`: 「このAIについて」→ 以下を箇条書き。
  - 入力: 登録した条件 + 登録済み式場 N 件
  - 除外: 登録済みは重複提案しません
  - 出典: 公開情報を元にした推定（予算は目安）
  - 更新: 1 時間ごと / 手動更新可
- 各カードに**根拠チップ**（最大 3 つ）: `エリア一致` `予算内` `料理重視` 等。`reason` 文章と併用。

### E. ユーザーコントロール

- 見出し右に `…` メニュー: 「24時間非表示」「AIおすすめをオフ」「フィードバック」。
- オフは `UserPreference.aiRecs = false` を追加（将来 schema、今は cookie 可）。
- カードに 👍 / 👎（任意、ログのみ）— 以後の精度改善フック。

## 5. 実装ロードマップ

| Phase | 内容 | 想定 Effort |
|---|---|---|
| P0（即時）| A のゲーティング + `return null` 廃止 + B の 5 状態文言 | 3h |
| P1 | C（SSR 初期状態 + CLS 固定高）+ D（根拠チップ & なぜ展開）| 0.5d |
| P2 | E（非表示 / オフ / フィードバック）+ Sentry 計測 | 0.5d |
| P3 | A/B: 推薦セクションありなしで D0 式場追加率を比較 | 1w 観測 |

## 6. コピー Before / After

| 状態 | Before | After |
|---|---|---|
| Loading | あなたの条件に合う式場を探しています... | おふたりの希望（都内・300万以内）で候補を整えています |
| Result 見出し | AIおすすめ式場 | あなたの希望に寄り添う 3 件 |
| Result sub | （なし） | 料理重視 / 300万以内 / 登録 2 件を除外 |
| Empty（失敗） | （消える） | 今の条件では新しい提案が見つかりませんでした |
| Primed | （消える） | 気になる式場を追加すると、AI が次の 1 件を提案します |
| Error | （消える） | いま少し混雑しています。少し後でまたお試しください |

## 7. 要意思決定

1. **ゲート閾値**: 提案は `venues>=1` 開始、`conditions` 0 項目時も許容するか？（推奨: 条件 1 つ以上で ON）
2. **閉じる**: セッション単位 or 24h 永続 or ユーザー設定？（推奨: 24h）
3. **根拠公開**: 「なぜ」展開をデフォルト開く / 閉じる？（推奨: 閉じる、但し初回は開く）
4. **失敗時プレゼンス**: エラーを見せるか無音にするか？（推奨: 見せる。沈黙が不信の原因）
5. **費用**: Claude 呼び出しを `venues>=1` に絞れば現状比で呼び出し減。ただし毎 explore 訪問で発火は過剰。**SWR 的に 24h キャッシュ + 式場追加/条件変更で invalidate** に変更するか？（推奨: YES）

## 参考（ベストプラクティス）

- **Spotify Daily Mix**: 履歴 0 件時は機能を出さず「まず 5 曲保存」の予告だけを出す（プレゼンス管理）。
- **Netflix**: 各推薦列の見出しに「◯◯を見たあなたへ」と**根拠を必ず明示**。
- **Amazon "Because you bought"**: 入力（購入品）を必ず引用。
- **Notion AI / Linear**: 失敗時は赤ではなく muted の一行説明＋再試行。
- **Airbnb "Based on your search"**: 条件チップと件数を見出し直下で可視化。

共通パターン: **(1) 入力を明示 (2) 根拠を添える (3) 沈黙で消さない (4) ユーザーに非表示の権利**。本計画はこの 4 点を Haretoki 文脈（Morning Light、細字、急かさないトーン）に落とし込む。
