# Wave 15 実装プラン: F1-F4 priority-A 不便ポイント解消

## Context
docs/designs/ に W15 の F1-F4 設計書（5 層 × design-reviewer 査定済み）が commit `a3924b6` (2026-04-22) で着地している。設計フェーズは完了、実装はまだ。memory snapshot 以降 114 commits 進んでいるため roadmap_state.md は更新が必要。本 plan は F1-F4 を 4 worktree 並列で実装し、Wave 15 を develop merge → prod deploy まで切る。

設計書で解消する痛み:
- F1: URL コピペ 3 ホップを削減（式場名検索で追加）
- F2: 見学日程の紙手帳/アプリ二重管理（.ics export）
- F3: 「決定で終わる」失速感（15 件定型 todo seed + /preparation 画面）
- F4: パートナーの「アカウント作るのだるい」離脱（Guest mode Level 1）

## Approach
**Foundation → 4 worktree 並列 → 統合 → Ship Cycle** を踏む。memory `agents.md` の N+1 tmux pane（チャット 1 + viewer 4）+ AgentTeams で進める。

設計書 reviewer の重要 blocker は実装時に必ず反映する:
- F1: server action 内 authz、Places session token、`projectId` 単位 rate-limit
- F2: METHOD:PUBLISH（Google auto-reply 回避）、VTIMEZONE 明示、`Visit.sequence` カラム必須、GET ハンドラから DB 書込を分離
- F3: `@@unique([projectId, templateKey])` で idempotent seed、別 venue 再決定時の completedAt reset、post-commit seed
- F4: `consumeInvitationLink` の `isAutoCreatedEmptyProject` guard、cookie に keyId/screenCount/lastSeenAt、secret 2 世代 rotation、`ProjectInvitation.lastViewedAt` + `viewCount`

## Foundation Phase（並列前に 1 人で完了させる）
場所: `develop` 直接 or `feat/w15-foundation` worktree。

   - `GUEST_SESSION_SECRET` / `GUEST_SESSION_SECRET_PREVIOUS`（F4 — 2 世代 rotation 用）

## Roll back / Risk
- F1 の Places API 月 cap 超過時は Claude fallback のみで degrade（hard gate なし）
- F4 の cookie secret 漏洩時は `GUEST_SESSION_SECRET_PREVIOUS` 経由で旧署名を一度だけ受け入れた後、新 secret に切替（rotation）
- F2 の `.ics` UID/SEQUENCE は仕様逸脱すると Google Calendar の「重複イベント」を呼ぶため、SEQUENCE は必ず DB 永続化
- F3 の seed は idempotent。`@@unique([projectId, templateKey])` 違反時は upsert で吸収

## 並列実行手順
1. Foundation を develop で完成 → push
2. tmux 現ウィンドウを `split-window -d` × 4 でチャット 1 + viewer 4 構成（memory `agents.md` N+1）
3. `TeamCreate` で 4 implementer team を起動。各 team は自分の worktree のみ触る
4. team 完了 → reviewer agent でセルフレビュー → fix まで完了させてから merge
5. すべて merge 後、E2E + Ship Cycle
