# Haretoki — あるべき全画面・全タップ・全遷移・全反応マップ

> 導出根拠: プロダクトの提供価値・目的・ターゲットユーザー・ユースケースから演繹。
> 既存設計書の正誤は問わず、「このプロダクトが約束する体験」から逆算した理想状態。
> 作成日: 2026-04-16

---

## 0. 導出の前提

### プロダクトの約束
「式場選びを、もっと納得のいくものに。」— 式場を「売る」メディアではなく、カップルの「選ぶ」を支援する中立ツール。

### ターゲットユーザー
20-30代カップル。こだわりたいが時間がない。ITリテラシー高め。右も左も分からないが、妥協はしたくない。

### ユースケース（感情の弧）
```
不安 → 発見 → 比較 → 確信 → 祝福
"何から始めれば?" → "こんな式場が!" → "データで納得" → "二人で決めた!" → "おめでとう!"
```

### 設計原則（プロダクトの約束から演繹）
1. **どの画面からでも自由にジャンプ可能** — 結婚式場選びは線形プロセスではない
2. **ステップ感・仕事感を出さない** — 管理ツールではなくプレミアム体験
3. **二人のプロダクト** — パートナーの視点が常に存在する
4. **AIは先回りで提案する** — ユーザーが聞く前に気づきを届ける
5. **全タップに即時フィードバック** — 150ms以内に何かが変わる
6. **行き止まりゼロ** — どの状態でも次のアクションが明確

### T3: 候補 (`/candidates`)

**あるべき構造:**
この画面は候補の管理・比較・決定を一手に担う中核画面。以下の要素が**常に**アクセス可能でなければならない。

#### 上部: メインSegmentedControl（常時表示・固定）
| セグメント | 役割 | 有効条件 |
|-----------|------|---------|
| 候補 | お気に入り式場の一覧 | 常に有効 |
| 比べる | 決定マトリクス（総合比較） | 候補2件以上 |
| 観点別 | 次元ごとの深掘り比較 | 候補2件以上 |

| パートナー | パートナーがお気に入りした式場 |
| おふたり | 両方がお気に入りした式場 |

**重要原則**: FavoriteFilterを切り替えても、上部のSegmentedControlは**常に見え、常にタップ可能**。フィルタはSegmentedControlの下に表示。

## 3. クロスカッティングなインタラクション（全画面共通）

### BottomNav（全画面で常時表示）
- 5タブ全てが**常にタップ可能**
- アクティブタブはgold-warmハイライト + ラベルfont-medium
- タブ遷移は300ms ease-out
- バッジ: 候補タブに候補件数、コーチタブに未読インサイト数
- 現在タブの再タップ: スクロールトップに戻る

### エラー状態
- Server Action失敗: Sonner toast + リトライボタン
- ページエラー: error.tsx（「エラーが発生しました」+ リトライ + ホームに戻る）
- AI応答遅延(10秒超): プログレスバー + 「AIが考え中です…」

### 空ステート
全ての空ステートにはCTAが必要（行き止まりゼロ原則）:
- 式場0件: 「式場を追加する」→ `/explore?addVenue=1`
- 候補0件: 「式場を探す」→ `/explore`
- 見積もり0件: 「見積もりを追加」→ Sheet
- 見学0件: 「見学を記録する」→ Sheet

### 戻る操作
- iOS back swipe: 全画面で機能
- ブラウザ戻るボタン: 全画面で機能
- 明示的戻るボタン: サブ画面（S1-S11）のヘッダー左に表示
