> 対象: `docs/myreview/problems_01.md` の実ユーザーフィードバック
> 作成日: 2026-04-14
> 前提: [DESIGN.md](../../DESIGN.md) v4 "Modern Luxury" / 4タブ構成（ホーム / 探す / 候補 / コーチ）
> スコープ: 情報設計・ビジュアル刷新・画面別改善・マイクロコピー・モーション

## エグゼクティブサマリ

ユーザーが突き付けた最大のメッセージは **「ボタンが多いのにプロダクトが語っていない」** である。
現状は「仕事感（タスクを並べた管理ツール）」が勝ち、ブランドの核である「曇り→晴れ間→晴れの日」という**感情の弧**が画面に出ていない。症状は折り返し崩れ・アフォーダンスの嘘・冗長ガイド・表記揺れといった個別バグに見えるが、根本は2つ:

1. **情報設計の破綻** — ホームの `JourneyCard` が「進捗ステップ表示」と「次の一歩NBAカード」を同じ箱に同居させた結果、"比較/追加/見学" の丸いアイコン（非リンク）と直下のCTA（実リンク）が視覚的に等価に見え、タップ判定が読めない。
2. **ビジュアル言語の古さ** — Morning Light トークンは正しいが、表層のレイアウトが「カード＋見出し＋リスト」の直列配置に留まり、2024-2026年のモダン・ラグジュアリー表現（非対称グリッド、gradient mesh、frosted glass、typographic contrast、editorial whitespace）を1つも採用していない。

本プランは、**情報設計の再分割 → ビジュアル言語の刷新 → 画面別の具体修正 → コピー統一 → 知覚速度の改善** の順で提示する。

## 1. IA（情報設計）再設計

### 1.1 タブ責務の再定義（決定版）

現状のユーザー混乱の核は「探す」タブに **探す・一覧・追加** の3つの仕事が同居していること、そしてホームの「すべて→」が「探す」に飛ぶため **一覧が見たいだけのユーザーが検索UIに放り込まれる** 不整合。

| タブ | 新・責務 | 旧・問題 | 主要動線 |
|---|---|---|---|
| **ホーム** | 今日の1アクションを提案する "Today" 画面。パーソナル+編集的 | 進捗/NBA/最近見た/CTAが詰め込まれすぎ | 1画面=1ヒーロー+1インサイト+最近見た |
| **探す** | **まだ候補にないものを見つける** 。検索・発見・追加の場所 | 「気になる式場を集める場所」という矛盾コピー | 検索バー / AIレコメンド / 追加 |
| **候補** | **集めた式場を育てる** 。一覧・評価・比較・決定の場所 | 旧「ショートリスト」のまま | 一覧グリッド / 比較ボード / 最終決定 |
| **コーチ** | AIとのセッション。相談の場所 | 過去チャットが入口に貼り付いている | セッションリスト / 新規チャット / インサイトフィード |

**重要な方針転換**: 「最近見た式場」の "すべて" は **候補タブに飛ばす** のではなく、**候補タブの「最近見た」セクション（新設）** へ遷移する。探すは発見の場所、候補は育てる場所という境界を崩さない。ホームから来たユーザーは「もう自分が見つけた式場たち」を見たいのであって、新規検索UIは求めていない。

### 1.3 ホームの情報構造 — 「押せないボタン」の解消

現状 `JourneyCard` の中で `JourneySteps`（追加/見学/比較/決定の丸いアイコン）と NBA（「次の一歩」CTAチップ）が同居している。丸いアイコンは非タップのステータス表示だが、ピル型バッジ + ゴールドのグロウがタップ可能に見える。**これがユーザーが "押せそうで押せない" と感じた直接原因。**

解消案は2段構えで:

**A案（推奨・低リスク）: タップ可能化してCTAと統合**
- 4つの丸アイコンを `Link` に変え、それぞれが実際に `/candidates?step=visit` 等の対応画面に遷移するディープリンクにする
- アイコン下の `現在/完了/未着手` ステータスは視覚的に保持（色・透明度で区別）
- NBA カードは残すが、文言を **「次の一歩」ではなく「おすすめ」** に変える（次の1件だけをサジェスト、ステップ一覧とは役割を完全に分離）

**B案（推奨・理想形）: JourneySteps を退場させ、Journey Ring + Hero NBA に一本化**
- 4ステップの水平インジケータを廃し、**1つの円形プログレスリング**（0-100%）＋ **現在のステージ名**（例: "見学中"）のみに圧縮
- その下にヒーローNBA（次のおすすめ1枚）を editorial カードとして置く
- ステップ別のディープジャンプはホームからは削除し、候補画面のサブタブで担う
- 根拠: 結婚式場選びは step N of 4 という線形でなく、実際は行ったり来たりする。ステップ棒は Asana 的 "仕事画面" に見え、ラグジュアリー文脈と最も衝突する要素

**最終推奨**: **B案**。DESIGN.md v4 原則「ステップ番号は見せない（仕事感の排除）」と「進捗は控えめなリング表示のみ」に既に明記済みで、現状実装がその原則に反している。つまり今回の指摘は **仕様通りに戻す** 作業。

新ホームの情報構造（上から下へ）:

```
1. Greeting (Noto Serif JP, 300, fluid-xl)
   「こんにちは、○○さん」
   サブ: 「晴れの日まで 127日」 (tabular-nums)

2. Hero NBA — Today's Weather Card  (唯一のヒーロー)
   左上: 天気アイコン (Cloud/CloudSun/Sun) = ステージ表現
   中央: 編集的コピー「今日は、2件目の見学予約を入れてみませんか」
   右: 円形プログレスリング (直径72px、数値なし、刻みのみ)
   主CTA 1つのみ: [この1件を入れる →]
   副CTA (弱): 別のおすすめを見る（text link）

3. Recent Venues (横スクロール、写真ファースト)
   セクションタイトル: 「先日ご覧になった式場」
   "すべて→" → /candidates?view=recent に飛ぶ

4. AI Insight (条件付きで出現、Gold subtle)
   週に1-2回しか出さない。毎日出すとノイズ化する

5. (ホームには他を置かない)
```

押せないボタン問題は **JourneySteps の削除で根治** する。「追加/見学/比較/決定」という単語をホームから消すことで、表記揺れ（「比較」vs「比べる」vs「比較する」）の混在源も同時に絶たれる。

## 2. ビジュアル刷新方針 — 2026 Modern Luxury

### 2.1 現状診断

Morning Light パレット（Cream #FAF8F3 / Rose #F8E9E4 / Gold #B88A4C）・Noto Serif JP・細字見出し・44pxターゲット・`active:scale-[0.98]` — これらは2026年水準として**正しい骨格**。問題は、その上に乗っている表層が「Bootstrap 時代の縦並びカード」であること。具体的欠落:

1. レイアウトが**単調な縦積み**（DESIGN.md は Bento を謳うが実装は space-y-16 の直列）
2. **色面が均一**（Cream ベタ + 白カード + gold subtle 1色、グラデーション一切なし）
3. **タイポのコントラストが弱い**（Noto Serif の細字は使っているが、巨大ディスプレイ数字 × 極小キャプションのような対比がない）
4. **モーションが装飾的**（ページイン 0.9秒のフェードは入れているが、遷移・状態変化・タップにキビキビ感がない）
5. **写真の扱いが素朴**（4:3トリム＋グラデオーバーレイのみ。Aspect非対称・マスク・パララックスなどの現代表現なし）

### 2.2 参考にする実在プロダクトのパターン

Refero を使わず、開発者として実物を触った記憶に限定して根拠を示す。

| プロダクト | 抽出パターン | Haretoki への転用 |
|---|---|---|
| **Airbnb (iOS, 2024-)** 式場ページ詳細 | 画像カルーセル上部に半透明ヘッダ、下部は editorial セクション + sticky 予約バー | 式場詳細のヘッダー/CTAをsticky化、下部タブ化 |
| **Aman / Aesop 公式** | 余白多めの editorial グリッド、明朝見出しの行間 1.4-1.6、句読点の逃げ | 見出しの tracking/leading を editorial 水準に |
| **Linear (2024-)** ホーム | Bento 2x3 + asymmetric weights、状態はドット1つで表現、重要情報だけ目立たせる | ホームHero + Recent + Insightの3ペインに圧縮 |
| **Arc Browser / Raycast** | frosted glass (backdrop-blur) + 極薄 border + inner shadow | Sheet・モーダル・ナビバーで採用 |
| **Superhuman** | 1アクションに1ホットキー、操作フィードバックが 100-150ms で完結 | タップ後の色/スケール変化を150ms厳守 |
| **Notion Calendar / Cron** | 日付を tabular-nums + 明朝の大文字ディスプレイ | 「127日」表記をディスプレイ扱いに |
| **ChatGPT / Claude モバイル** | 左スワイプでセッション一覧ドロワー、時系列グルーピング | コーチ画面の履歴UI |
| **Partiful** | プレイフルな gradient mesh + グラスモーフィズム + タイポのジャンプ率 | ランディング・空状態・お祝い画面（決定後）で部分採用 |
| **Zola (wedding app)** | 進捗は%ではなく「So far so good」的な感情コピーで表現 | NBA 文言を感情ドリブンに |
| **The Knot** | 式場カードは写真65%+情報35%、価格はゴールド強調 | 現状実装と合致、維持 |

### 2.3 Haretoki 向け 2026 ビジュアル方針（5本柱）

**1. Editorial Rhythm（雑誌的リズム）**
- セクション間は機械的な `space-y-16` をやめる。Hero下は96px、secondaryは64px、tertiaryは40pxの**非対称垂直リズム**
- Hero見出しは `text-fluid-xl`（clamp 28-36px）、サブは 13px muted。**中間サイズを禁止**してジャンプ率を強制
- 本文 Noto Sans JP は leading 1.8、見出し Noto Serif JP は leading 1.35、tracking 0.02em

**2. Atmospheric Color（空気感のある色面）**
- 新トークンを3つ追加: `--gradient-dawn`（Cream → Rose 微グラデ、radial）、`--gradient-noon`（Rose → Gold 微グラデ、linear 45度）、`--gradient-dusk`（Gold → Cream 微グラデ、ホームHero背景）
- 濃度は全て 4-8% のみ。ベタの上に乗せる **空気の層** として使う
- ダークモードは Dawn/Noon/Dusk を Ink Blue / Wine Rose / Amber に置換

**3. Material Layer（素材のレイヤ）**
- frosted glass: `bg-white/60 backdrop-blur-xl border-white/40` をボトムナビ・Sheet・stickyヘッダに
- inner glow: Hero カードに `shadow-[inset_0_1px_0_rgba(255,255,255,0.6)]` を足し、エンボス感
- gold hairline: 重要区切りに 0.5px gold-warm ライン（現状のborderは均質すぎる）

**4. Typographic Contrast**
- 「あと127日」の **127** は `font-serif`・`text-5xl`・`tabular-nums`・`tracking-tight` で **ディスプレイ扱い**、「日」は 12px サンセリフ
- 同じ原則を予算「¥350」万、見学件数、進捗% に適用
- 式場名は Noto Serif JP 300、サブ情報は Noto Sans JP 400 という**ウェイトと書体の二重対比**

**5. Micro-Motion Discipline**
- 入場 600-900ms（ラグジュアリー感）、操作反応 **150ms 厳守**（Superhuman 水準）
- Spring: stiffness 180 / damping 14（DESIGN.md v4 既定通り）
- 新モーション: **Halo Tap** — タップ時に gold ring が 1回だけ広がって消える（250ms）。主要CTAのみ
- View Transitions API を list→detail 遷移で採用（Chrome/Safari 対応済）

### 2.4 DESIGN.md 具体アップデート案（差分）

- セクション「Modern Luxury UX Principles」の後ろに **「Atmospheric Layers v4.1」** を追加:
  - 新カラートークン3つ（gradient-dawn/noon/dusk）の定義と用途マップ
  - Material Layer 規定（frosted glass の blur強度/opacity、inner glow の値、gold hairline の使用条件）
- 「Home (ホーム)」節を書き換え: Bento 2x3 構造を **Hero + Recent + Insight の editorial 縦積み** に修正。JourneyCardは廃止、Journey Ring に置換
- 「Explore (探す)」節: ヘッダーコピーの修正 + FAB化した「追加」ボタンの仕様追加
- 「Coach (コーチ)」節: **Sessions** セクションを新設（ChatGPT式履歴ドロワー）
- 新セクション「Microcopy Lexicon」: 表記揺れを公式に固定（下記第4章）
- 新セクション「Perceived Speed Rules」: prefetch / Optimistic UI / skeleton / View Transitions の具体規約

## 3. 画面別改善プラン

### 3.1 ホーム

#### H-1: 「お二人の式場探し」が「し」だけ改行される
- **問題**: `JourneyCard` 見出し「おふたりの式場さがし」が 375px幅で font-serif 300/24px、右に天気バッジを置いた結果、残幅が足りず最終文字だけが折り返す
- **改善**:
  - 見出しを「おふたりの式場さがし」→ **「おふたりの一年」** または **「ふたりの旅程」** など **Journey Ring に沿った短文** に置換（6文字以内）
  - または Journey Card 自体を廃止する B案を採用すれば見出し自体が消える
  - 折り返し制御は `text-wrap: balance` + `inline-block` + 天気バッジを**見出しの上**に移動して幅を取り切る
- **参考**: Airbnb の「Wishlists」的な1語見出し、Aman の「Your Stay」
- **難易度**: S（B案採用なら自動解決）

#### H-2: 押せそうで押せないボタン（比較/追加/見学）
- **問題**: `JourneySteps` のアイコン円がタップ判定されず、視覚的アフォーダンスと挙動が矛盾
- **改善**: B案で `JourneySteps` を削除し、Journey Ring + Hero NBA に置換（1.3 参照）
- **参考**: Linear ホームの "inbox/projects/active" クリッカブルチップ
- **難易度**: M（コンポーネント削除 + Ring 新設）

#### H-3: 「次の一歩」と「比較してみましょうガイド」の重複
- **問題**: 同じメッセージが2箇所。`JourneyCard.NBA` と、別コンポーネント（AIインサイトカード or ホーム下部のガイド）が同時出現
- **改善**:
  - ホームでは **1画面=1 NBA** の鉄則。Hero NBA のみを残し、他のガイド/インサイトは週1-2回の登場制限
  - NBA と AIInsightCard が同じ行動を促す場合は、**AIInsightCard を非表示** にする優先順位ロジックをサーバー側に
- **参考**: Superhuman の "next email" 単一フォーカス、Notion Calendar の "Today at a glance"
- **難易度**: S（条件分岐追加）

#### H-4: 「比較する」押下→「うまくいきませんでした」のバグ
- **問題**: NBA の「比べる」リンクが `/candidates` に飛ぶが、候補画面自体がエラー状態
- **改善**: これは情報設計ではなく **候補画面のサーバー実装バグ**。デザイン観点では、リンク先が未整備の状態で CTA を露出しない **guard** を入れる（favorite≥2 かつ候補画面が正常応答する時のみ表示）
- **難易度**: M（原因は別タスクだが UI からは短絡防止可能）

#### H-5: 「すべて→」が探す画面へ飛ぶ
- **問題**: 意味的に不整合。最近見た ≠ これから探す
- **改善**: リンク先を **`/candidates?view=recent`** に変更。候補画面に「最近見た」サブタブを新設
- **参考**: Airbnb の「Recently viewed」は Trips/Wishlists 配下にある
- **難易度**: S（リンク先変更 + 候補画面にフィルタ追加）

### 3.2 探す

#### E-1: 追加ボタンが地味
- **問題**: `AddVenueSheetTrigger` が `Button size="sm"` の標準スタイル
- **改善**:
  - 右上の「追加」を削除し、**右下 FAB (Floating Action Button)** に昇格
  - FAB 仕様: 56×56px、gold-warm 背景、白 Plus アイコン、`shadow-elevated`、`env(safe-area-inset-bottom)` に16pxマージン
  - スクロール下方向でフェード、上方向で復活（Material 3 の extended FAB 相当）
  - ヘッダーの「追加」は残すが、テキストリンク程度の控えめ扱いに
- **参考**: Airbnb ホストモード、Notion モバイル新規作成
- **難易度**: S

#### E-2: 「探す」と「気になる式場を集める場所」の矛盾
- **問題**: タイトルと副題がMECEでない
- **改善**: 副題を「**まだ見ぬ式場を、見つける**」に変更。「集める」は候補タブの仕事
- **難易度**: S

#### E-3: 式場追加モーダル — タブ構成がMECEでない
- **問題**: 「自分で / URL / まとめて」の並びが同じ粒度でない（URLは単数・手動は単数・まとめては複数）
- **改善**: **2段構造に再編**
  - **上段（Primary）**: URL貼り付けエリアを大きく常設。1件でも10件でも同じボックスに複数行貼れる（改行で分割）。URL貼り付けが**最速＆最も使われる**経路であることを視覚的に表現
  - **下段（Secondary、折りたたみ）**: 「URLがない場合は手動で追加」リンク → タップで下に手動フォームが展開
  - タブUIを廃止。**線形な1フォーム**にする
- **参考**: Raycast の "Quick Add"、Notion の "Import from URL"、Pinterest の "Save from URL"
- **難易度**: M（コンポーネント再構築）

#### E-4: URL追加失敗時のトースト永続化 + タブ戻り時に入力残る
- **問題**: トーストが消えない + 手動タブに自動遷移するが URL タブに戻すと前の URL が残る
- **改善**:
  - トーストは **タップ可能な Sonner Toast** に変更（タップで即消し、3秒自動消滅）
  - AI解析失敗時は**タブ自動切り替えをやめる**。代わりに URL 入力欄の下に「自動で読めませんでした。**手動で入力する →**」というインラインCTAを出す。これなら URL が残っていても違和感ない
  - タブ切り替え時に前タブの state を `reset()` する（ただしE-3の再編で解決）
- **難易度**: S

#### E-5: 式場追加モーダルが「ぜんぜんわくわくしない」
- **問題**: グレーのフォーム + 文字だけ
- **改善**: Sheet の上部に **editorial ヘッダー** を追加
  - 左: `font-serif text-fluid-xl font-extralight` で「新しい式場を、迎える」
  - 下: 12px muted で「URLを貼るだけで、AIが情報をお持ちします」
  - 背景: `--gradient-dawn` を sheet 上部 30% にradialで薄く入れる
  - URL入力後の「読み取り中」を、スピナーではなく **skeleton + 段階的に項目が埋まる** 演出に（式場名 → 住所 → アクセス → 写真 の順でフェードイン）
- **参考**: Partiful の "Create event"、Arc の "New space"
- **難易度**: M

#### E-6: AIおすすめ式場が「今少し混雑しています」のまま
- **問題**: 初期seed が `shouldRequest` を常に true にして負荷超過、または seed fetcher 実装バグ
- **改善**: デザイン観点では**フォールバックUIを editorial に**
  - エラー時は「少し混雑しています」ではなく、**「今日のAIおすすめは準備中です。先日ご覧になった式場をどうぞ →」**
  - リンクで Recent Venues にスクロール。**行き止まりを作らない**
- **難易度**: S

### 3.3 式場詳細

#### V-1: ラベル書式が不統一（施設名・住所・アクセス）
- **問題**: `VenueHeader` が各フィールドを異なる視覚階層で並べている
- **改善**: **定義リスト（definition list）パターン**に統一
  - ラベル側: 11px/muted/tracking-wide/uppercase-equivalent（Noto Sans JP 500）
  - 値側: 14px/foreground/leading-1.7
  - ラベル幅を固定（80px）して縦のグリッド感を出す
- **参考**: Aesop 商品詳細の "Category / Skin Type / Size"、Aman の部屋詳細
- **難易度**: S

#### V-2: 縦長UIをスライド/タブに分割
- **問題**: Header → Photo → Rating → Estimate → Review → Plan → Visit → X-Ray → Waterfall がひたすら縦並び
- **改善**: **Sticky セグメント化**
  - 上部: Photo Gallery（既存）+ Header + 主要CTA バー
  - その下に **Sticky Segmented Control**: `[概要] [見積] [見学] [口コミ] [AI解析]`
  - 各セグメント選択でスムーススクロール + 対応アンカーへ移動（IntersectionObserver で双方向同期）
  - スクロールで自動的にアクティブセグメントが切り替わる（Airbnb 予約詳細の実装パターン）
- **参考**: Airbnb listing の "Amenities/Location/Reviews" sticky tabs、Apple Music アルバムページ
- **難易度**: L（scroll-spy 実装 + レイアウト再構築）

#### V-3: 見積もり項目名のプリセット化
- **問題**: 現状は自由入力のみ
- **改善**: **Combobox パターン**
  - 入力欄をクリックでプリセット40種（料理、飲物、ウェディングケーキ、衣装、ブーケ、装花、ヘアメイク、写真、映像、司会、演奏、引出物、ペーパーアイテム、会場使用料、サービス料…）をドロップダウン表示
  - タイプするとインクリメンタル絞り込み
  - 該当ない場合は **「"◯◯" を自由入力で使う」** という最下行候補
  - プリセットに属するとカテゴリ自動付与（料理→飲食、写真→撮影）
- **参考**: Notion の "Select or create"、Linear の label picker
- **難易度**: M

#### V-4: 「他のカップルも同程度の調整をしています」が2回表示
- **問題**: `EstimateXRay` と `EstimateWaterfallChart` の両方に同じコピーが重複
- **改善**: Waterfall 側のみ残し、X-Ray 側のベネフィットコピーは **「見積の差分を、項目ごとに」** 等、機能説明に変更。同じ安心メッセージを二度言わない
- **難易度**: S

#### V-5: 星評価0.5刻み + 評価保存失敗
- **問題**: 0.5刻み未対応 + Server Action の保存失敗
- **改善**:
  - ドラッグ可能な水平バーにしてピッチ0.5を視覚的に強制（Airbnb 評価UIに近い）
  - 保存失敗時は楽観的更新＋ロールバック＋「もう一度お試しください」（具体エラー）
- **難易度**: M

### 3.4 口コミ

#### R-1: 「気になる点を先に」ボタン
- **問題**: 意味不明。ユーザーは何が起こるか推測不能
- **改善**:
  - ボタンを削除し、代わりに **Review Sort Chip**「並び替え: 最新 / 評価高い / **気になる点から**」に統合
  - 「気になる点から」を選ぶと、ネガティブ要素を含むレビューが上位に並ぶ（ユーザーの "落とし穴を知りたい" 意図に正直）
- **参考**: Booking.com の "Show reviews with concerns first"、Amazon の "Critical reviews"
- **難易度**: S

### 3.5 コーチ

#### C-1: 過去チャットが入口に貼り付いている
- **問題**: コーチ画面を開くと前回の会話がそのまま見えていて、新規質問しにくい
- **改善**: **ChatGPT/Claude 式セッション管理**
  - 入口は **新規チャット画面**（空のチャットバー + クイックスタートプロンプト + 直近インサイト3件）
  - 左上に **履歴アイコン**。タップで **Sheet from left** が展開し、セッション一覧
  - セッションは `今日 / 昨日 / 今週 / それ以前` でグルーピング（ChatGPT 方式）
  - 各セッションは最初のユーザー発話20字でタイトル化（自動）、長押しで手動リネーム可
  - サーバー側で `CoachSession` エンティティ新設（projectId, title, createdAt, updatedAt）、`CoachMessage` が `sessionId` を持つ
- **参考**: ChatGPT iOS、Claude iOS、Perplexity history drawer
- **難易度**: L（データモデル変更 + UI + マイグレーション）

#### C-2: AIインサイトフィードとチャットの関係
- **問題**: 現状、コーチ画面に Insight カード + 過去履歴 + チャットバーが全部乗っている
- **改善**: **サブタブで分離**
  - トップ: セグメント `[新規チャット] [インサイト]`
  - 新規チャット: 空入力状態 + 履歴ドロワー呼び出し
  - インサイト: 現状のカードフィードを維持（週1-2件のペースに制限）
- **難易度**: M

### 4.1 動詞の揺れ解消

| 文脈 | 統一後 | 禁止表記 |
|---|---|---|
| 候補の式場を横並び評価する行為 | **「比べる」** | 比較する / 比較してみる |
| 式場の情報を並置して見るUI | **「比較ボード」** | 比較画面 / 比べるページ |
| 式場を候補に入れる行為 | **「候補に入れる」** | お気に入り / ショートリスト / ブックマーク |

## 5. モーション・遷移体験（Perceived Speed）

「画面遷移が遅すぎる」というフィードバックは、実測速度よりも **知覚速度** の問題である（Server Actions + Supabase はそもそも100-300msで返る）。下記4本柱で改善:

### 5.1 Prefetch 徹底

- 全 `<Link>` に `prefetch={true}` を明示（Next.js 16 App Router のデフォルトは条件付きprefetch）
- ホームのRecent Venuesカード、候補一覧カード、コーチ履歴項目は **viewport prefetch**: IntersectionObserver で50%表示された時に詳細ページをプリフェッチ
- ボトムナビの4タブは **mount時に即prefetch**（初回レンダ後100ms遅延で）

### 5.2 Optimistic UI 全面採用

- 候補追加/解除のハート: クリック直後にUI変化 → Server Action → 失敗時ロールバック + トースト
- 評価更新: ドラッグ中はローカルstate、離した瞬間にサーバー送信
- チャット送信: メッセージを楽観的に即表示、送信中はopacity 0.6、失敗時は再送ボタン
- 見積追加: 行を即挿入（skeleton風）、サーバー確定で実体化

React 19 `useOptimistic` フックで実装。DESIGN.md v4 原則 P3「フィードバックは即時かつ複層」を全Server Action呼び出しで徹底。

### 5.3 Skeleton と Layout Stability

- `loading.tsx` は**実レイアウトと完全一致**（高さ・角丸・余白）させる。現状の loading.tsx は薄いグレーブロックのみで、実際のカードとサイズがずれてレイアウトシフトが発生している
- skeleton 実装: `animate-pulse` ではなく **shimmer gradient**（左→右にgold-subtle が流れる）で、Morning Light 世界観を保つ
- 画像は `next/image` の `placeholder="blur"` + `blurDataURL` を全venue photoで使用

### 5.4 View Transitions API

- list → detail 遷移で画像を共有要素として滑らかに移動
- Chrome/Safari はネイティブ対応、Firefox は即時遷移フォールバック
- Next.js 16 App Router で `unstable_ViewTransition` が使える（実装時は stable 待ち）
- 具体対象: Explore カード → 式場詳細の4:3写真が **ヒーロー画像** として拡大しながら繋がる

### 5.5 Touch Response 150ms 厳守

- タップから150ms以内に**何かが動く**: 色変化 + scale(0.98) + inset shadow を同時発火
- 主要CTAに **Halo Tap** アニメーション（gold ring が1回広がって消える、250ms）
- DESIGN.md v4 既定の Spring stiffness 180 / damping 14 を徹底

### 5.6 ページ入場アニメーション

- Hero: 600ms（現状900ms → 短縮）、cubic-bezier(0.16, 1, 0.3, 1)
- Secondary セクション: 400ms、Hero から50ms stagger
- リスト項目: 個別ではなく**まとめて1回**フェード（過剰stagger回避、Aesop の節度）

### 5.7 計測と予算

- LCP 1.5s / FID 50ms / CLS 0.05 を予算化（[nonfunctional-requirements.md](../superpowers/specs/2026-04-13-nonfunctional-requirements.md) 参照）
- Vercel Analytics + Web Vitals で継続監視
- 遷移体感は Lighthouse ではなく **実機 iOS Safari + 3G throttle** で月1回検証

## 7. 完了判定（Definition of Done）

このプランが成功したと言えるのは、次のユーザー主観指標が揃った時:

1. **「押せるかどうか」を迷う瞬間がゼロ** — アイコンは全て Link か、Link でないなら装飾（透明度で明示）
2. **「同じことを2回言われた」感がゼロ** — ガイド・NBA・インサイトの責務が明確に分離
3. **「2026年のアプリだ」と口に出して言える** — editorial rhythm + atmospheric color + micro-motion で、2020年代中盤の水準に到達
4. **遷移で待たされる感覚がない** — 実測 <200ms、知覚 <100ms（optimistic + prefetch + skeleton）
5. **「比較する/比べる」等の揺れが全消** — Lexicon に全コピーが準拠

ユーザーが最初に言った **「20年前のデザインみたい」** から **「雑誌のアプリ版みたい」** に評価が変わる地点が、このプランのゴールである。ブランドメタファー（曇り→晴れ間→晴れの日）を、ステップ棒ではなく**画面全体の空気感（色・余白・余韻）**で表現するのが、モダン・ラグジュアリーへの唯一の道筋となる。
