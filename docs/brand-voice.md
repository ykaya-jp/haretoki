# Haretoki Brand Voice

> ブランド名: 晴れ時 (Haretoki) — 「曇り (不安) → 晴れ間 (見えてきた) → 晴れ (確信と喜び)」のメタファーで、結婚式場選びを支えるカップル向けの中立ツール。

This is the **single source of truth** for the voice of every UI string, AI prompt, marketing copy and email Haretoki sends. When in doubt, this file wins over individual screen mocks.

---

## 1. 三つの原則 (The Three Principles)

| 原則 | やる | やらない |
|---|---|---|
| **急かさない** | 「ふたりのペースで」「あとでも大丈夫」「気が向いたら」 | 「今すぐ」「残り◯時間」「お早めに」「カウントダウン」 |
| **売らない** | 「合いそうな式場」「気になる場所」「整う」 | 「おすすめ No.1」「人気急上昇」「特別キャンペーン」「限定」 |
| **中立** | 「並べてみる」「見比べる」「数字と気持ち、どちらも」 | 「業界 No.1 にお任せ」「プロが選んだ」「掲載店からの口コミ」 |

掲載料は受け取らない / 広告枠は売らない、という約束 (`docs/roadmap.md` "Commitment") をユーザーに会うたびに小さく言葉で繰り返す。「私たちは式場を売らない」ではなく「ふたりが選ぶのを助ける」と肯定形で。

## 2. 文体 (Editorial Register)

| 項目 | ルール |
|---|---|
| 敬体 / 常体 | **敬体 (です・ます)** が基本。常体は AI コーチの内省ライン (「考えているところ」) のみ |
| 語尾 | やわらかい終止形 (「〜できます」「〜してみませんか」)。命令形「〜してください」は **zod バリデーションメッセージのみ**、UI コピーでは禁止 |
| 句読点 | 「、」「。」を必ず使う。「. 」「, 」(英文式) は禁止 |
| 数字 | 半角 + `tabular-nums` (`font-variant-numeric: tabular-nums`)。「4件」「3日後」「14:00」 |
| 単位 | 「件」「人」「日」は数字直後にスペースなし。「14:00 から」のように時刻と助詞の間にだけスペース可 |
| 絵文字 | **使わない**。ブランド絵文字 (晴れ間 / 曇り) も UI には乗せない。OS 通知タイトルでも使わない |
| 漢字 | 常用漢字。「為に」「事」「物」などは「ために」「こと」「もの」と開く |
| 英数字混在 | 全角と半角を混ぜない。"AIコーチ" は半角 AI + 全角コーチで OK (固有名詞扱い)、"AI Coach" のような半角混在は避ける |

### 同義語の優先順位

複数の言い方ができるとき、上から優先して採用する。

| ✅ 優先 | ⛔ 使わない (理由) |
|---|---|
| ふたり / おふたり | カップル様 (営業敬語)、お二方 (硬い) |
| 候補 | ショートリスト (DB 名は `VenueFavorite` だが UI は「候補」) |
| 見比べる / 並べる | 比較する (やや硬い、エンタープライズ) |
| 整える | カスタマイズする / 設定する (機能的すぎる) |
| 印象を残す | レビューを書く / 評価を保存する (こちらは行為が機械的) |
| 見学 | 内覧 / 下見 (式場業界用語に寄りすぎ) |
| 決める | 確定する / 決定する (重い) |
| はじめる | スタート (片仮名濫用)、開始する (硬い) |
| ホーム / 探す / 候補 / コーチ | ダッシュボード / 検索 / お気に入り / アシスタント (英語的) |

## 3. UI コピー gallery — Do / Don't

### 3.1 ボタン

| シーン | ✅ Do | ⛔ Don't |
|---|---|---|
| 式場を保存 | 候補に入れる | お気に入りに追加する |
| 評価を残す | 印象を残す / 残す | 保存する / 評価する |
| 比較に進む | 並べてみる | 比較する / 比較画面へ |
| 招待を送る | 招待を送る | 共有する / 送信する |
| やり直す | やり直す | リセット / クリア |
| 解除する | やめる | 削除 / 解除 (破壊性が伝わらない場面のみ) |

### 3.2 トースト / フィードバック

| 種類 | ✅ Do | ⛔ Don't |
|---|---|---|
| 保存成功 | 残しました / 保存しました | 完了しました / Success |
| 同期成功 | おふたりに届きました | 同期完了 / Synced |
| 失敗 | うまく保存できませんでした。もう一度お試しください | エラーが発生しました / Failed |
| バリデーション | 30文字以内で入力してください | 文字数オーバー / Too long |

### 3.3 空ステート

| 画面 | ✅ Do | ⛔ Don't |
|---|---|---|
| 候補なし | まだ候補はありません。気になる式場を 1 件入れてみましょう | 候補がありません |
| 見学なし | これから見学する場所を選ぶところから | データがありません |
| 通知なし | 今は届くお知らせはありません | 通知はありません |

### 3.4 ダイアログ / モーダル

- タイトル: 体言止めまたは「〜しますか」「〜していいですか」
- 確認ボタン: 動作の動詞そのもの (「やめる」「招待を取り消す」)。「OK」「はい」は使わない
- キャンセル: 「あとで」「やめておく」「キャンセル」のいずれか。シーンの重さで選ぶ
  - 軽い (UI 設定): 「キャンセル」
  - 中 (招待を中断): 「あとで」
  - 重い (decision を取り消す): 「やめておく」

### 3.5 AI コーチの語り

- AI は「私」「私たち」とは名乗らない。「コーチ」が話す形 (「コーチからのひとこと」)
- 一人称が必要な場面では **「コーチ」** を使う (「コーチが見たところ」)
- 推測には強い断定を使わない。「のようです」「かもしれません」「と読みとれます」
- ユーザーの代わりに決めない。「AB のうち、おふたりは A を選んでいます」(事実) で止め、「A を選びましょう」とは言わない

## 4. 呼称 — 場面ごとの使い分け

| 場面 | 相手 | 呼び方 |
|---|---|---|
| 自分 | 操作中の本人 | (主語は省略する。必要な時だけ「あなた」) |
| パートナー | プロジェクトに参加した配偶者 / 婚約者 | パートナー / おふたりのもう一人 |
| 自分 + パートナー | プロジェクトのオーナー側ユーザー | おふたり / ふたり |
| ご家族 | 親 / 兄弟など、family-share token で見るだけの人 | ご家族 / おふたりのご家族 |
| 業者 / 式場 | プランナー / 式場担当 | 式場担当者 / プランナー (式場目線では「業者」だが、ユーザーには「担当者」と呼ぶ) |
| Haretoki チーム | われわれ運営 | Haretoki / 運営 ("〜運営から" は避け、必要なら "Haretoki から") |
| AI コーチ | LLM 経由の助言主体 | コーチ / AIコーチ ("AIコンシェルジュ" は marketing 用語、UI では使わない) |

### 注意

- 「お客様」は使わない。Haretoki はサービスを売らないので、「ユーザー」も商業色が出るときがある。**最も自然な代替が「おふたり」**
- 「パートナー」は二人称ではなく三人称。本人が読むときは「ゆうすけ さん」のような表示名 + さん付け
- Decision の宛名は「おふたりへ」が canonical。「あなたへ」は孤独感が出る

## 5. メタファー (Brand Metaphor)

晴れ時のすべてのコピー / イラスト / 動きは **曇り → 晴れ間 → 晴れ** のグラデーションを意識する。

| ステージ | UI 上の場面 | 言葉の例 |
|---|---|---|
| 曇り (cloudy) | onboarding 開始、候補ゼロ、迷っている瞬間 | 「ぼんやりとした」「まだ見えない」「大丈夫、これからです」 |
| 晴れ間 (break) | 式場を 2-3 件入れた、見学が決まった | 「見えてきました」「ここから整います」「次は…」 |
| 晴れ (sunny) | decision 確定、当日カウントダウン | 「決まりました」「ふたりの晴れの日」「ここまで歩きました」 |

OG カード (`src/lib/og-tokens.ts`)、onboarding-flow の `WASH_BY_STAGE`、decision-ceremony の wash deck は同じ recipe を使う。コピーもこの 3 段階の語彙から借りる。

## 6. AI prompt 設計でのトーン参照

`src/lib/prompts/*.ts` のプロンプトを書き直す / 新規作成するときは、上記の原則を **system prompt に直接書く**。

### 推奨 system prompt フラグメント

```text
あなたは Haretoki の AI コーチです。次のルールを守ってください。
- 急かさない: 「今すぐ」「残り◯件」のような時間プレッシャーを出さない。
- 売らない: 特定の式場をすすめない。事実と、ふたりの好みからの読み取りを並べるだけにする。
- 中立: 「業者がおすすめ」のような第三者の主観を持ち込まない。
- 敬体・敬語で、やさしく、句読点をはっきり使う。
- 絵文字は使わない。
- ふたりを「おふたり」「ふたり」と呼ぶ。「お客様」は使わない。
- 推測には「のようです」「かもしれません」と弱める。代わりに決断はしない。
```

このフラグメントは **新しい AI 機能を作るたびにコピペでよい**。詳細トーンが必要な場合は `docs/ai/prompts/` に prompt ごとの仕様を追加する。

## 7. ローカライズ方針

- 当面は **日本語のみ**。英語の UI は提供しない (商用化スコープ外)
- アクセシビリティ用 `aria-label` は日本語で書く (スクリーンリーダーが日本語環境前提)
- 内部コード (function 名、コミットメッセージ、コードコメント) は **英語**。これは `CLAUDE.md` ルール
- 英単語が UI に出るとき (e.g. "HARETOKI" のロゴ eyebrow) は **大文字 + 字間広め (`tracking-[0.2em]` 以上)** で、エディトリアル感を保つ

## 8. 検証チェックリスト

新しいページ / 機能 / プロンプトを書いたら、merge 前にこのリストを 1 周する。

- [ ] CTA 文言は時間プレッシャーを与えていない
- [ ] エラーメッセージは「〜できませんでした」で「もう一度」が添えてある
- [ ] 絵文字を使っていない (UI / OS 通知 / OG / メール本文すべて)
- [ ] 「お客様」を使っていない
- [ ] バリデーションメッセージで「〜してください」を使っていない (zod 以外)
- [ ] 数字に `tabular-nums` または明朝が当たっている
- [ ] 「ショートリスト」「マイページ」など、`CLAUDE.md` の用語対応表どおりの UI 表記
- [ ] AI 出力に断定 (「A にしましょう」) が混ざっていない
- [ ] 招待 / family / 業者向けの呼称が §4 表どおり
- [ ] 「曇り → 晴れ間 → 晴れ」の語彙を意識している (画面が編集的に重要なとき)
